# Verse Coding Conventions & Code Review Standards — Teravision Games

> **Source of truth for code reviews.** Both human reviewers and the AI pre-review use this document.
> When the AI flags an issue, it cites the section name in brackets — e.g. `[Naming]`, `[Concurrency]`.
> Add new conventions here when the team agrees on them. Versioned in the repo like any other code.

We follow Epic's [Verse Code Style](https://dev.epicgames.com/documentation/en-us/uefn/verse-code-style-guide-in-unreal-editor-for-fortnite) as the baseline. This document adds Teravision-specific rules and emphasizes the ones we deem most important. **If a topic isn't here but is in Epic's standards, follow Epic's standards.**

---

## Review philosophy

A code review is a teaching moment, not a gate to pass. Every comment should:

1. **Be specific.** Cite the line, the rule violated, and the impact. "This could deadlock if Player leaves during the spawn" — not "concurrency issue".
2. **Explain WHY.** Don't just say what to change; explain the reasoning. The author should learn something useful that applies beyond this one snippet.
3. **Suggest, don't demand.** Frame as "Consider X because Y" rather than "Change to X". Leave room for the author's expertise.
4. **Prioritize honestly.** Not every issue blocks merge. Use the severity markers below.
5. **Praise good code.** Call out clever solutions and clean patterns. Reviews that only criticize make people defensive.

### Severity markers (used by AI and humans)

The AI emits issues with one of these emojis. Reviewers use the same vocabulary:

- 🔴 **Blocker** — must fix before merge. Bug, security risk, data loss, broken concurrency, breaking API contract.
- 🟡 **Suggestion** — should fix. Maintainability problem, missing input validation, performance smell, missing test for important path.
- 💭 **Nit** — nice to have. Style nuance not caught by linter, alternative approach, naming improvement.
- ✨ **Praise** — explicit recognition of a good pattern. Reviewers and AI should add these when warranted.

### Comment format the AI follows

```
🟡 [Concurrency] Loop without yield will starve other tasks.
   Why: Aggressive loops consume CPU every frame and prevent other suspends from running.
   Suggestion: Add Sleep(0.0) at minimum, or convert to event-driven pattern using Subscribe.
```

Each issue has a section tag (cross-reference to this doc), severity emoji, brief title, "Why:" explanation, and a "Suggestion:" with concrete action.

## [Naming] Naming conventions

### Type names → `lower_snake_case`

Includes structs, classes, typedefs, traits/interfaces, enums. Always specify the kind in the name (`device`, `component`, `interface`, etc.):

```verse
clash_room_enemy_device
clash_defensive_structure_activator_device
mesh_component
agent_group_interface
```

### Everything else → `PascalCase`

Modules, member variables, parameters, methods, locals, constants:

```verse
var RoomTag : ?tag = false
ActivateByForce():void = ...
set CurrentEnemies = 0
```

### Booleans read like questions

No `b` prefix. Names should sound like a yes/no question:

```verse
# Good
IsAlive
IsComplete
InventoryIsEmpty

# Bad
bIsAlive
Alive
Empty
```

### Avoid abbreviations

Prefer legibility over brevity. `class_and_team_selector_device` not `cts_dev`.
For known acronyms use uppercase (`XP`, `ID`); for non-universal acronyms spell them out.

### Events vs event handlers — different naming

- **Events** (the thing) → no `On` prefix: `PlayerConnectedEvent`
- **Handlers** (the function that subscribes) → `On` prefix: `OnPlayerConnected()`

This avoids naming collisions in the same class.

## [Formatting] Indentation and spacing

### 4 spaces, never tabs

In `settings.json`:
```json
"[verse]": {
    "editor.insertSpaces": true,
    "editor.tabSize": 4
}
```

### Spaces around operators

```verse
# Good
Logger:log = log{Channel := log_crash_room_enemy_device}
NewAge := 24 + (2 * (5))

# Bad
Logger:log=log{Channel :=log_crash_room_enemy_device}
NewAge:= 24+ (2*(5))
```

## [Separators] Visual separators between methods

Place a separator above each method, with one blank line above the separator:

```verse
    #-----------------------------------------------------------------------------------------
    OnBegin<override>()<suspends>:void =
        if (DebugMode?):
            Print("Initial setup")

    #-----------------------------------------------------------------------------------------
    Activate():void =
        Logger.Print("CurrentEnemies {CurrentEnemies}")
```

Use them **consistently** within a file.

---

## [Comments] Comment style

- Add separators above methods.
- Add descriptive comments above methods **only when needed**.
- Always add a header comment summarizing the device/script/class.
- Skip comments for self-explanatory methods.
- For multi-line comments (3+ lines), prefer `<# ... #>`.

```verse
#-----------------------------------------------------------------------------
#   Device for items that can be collected.
#-----------------------------------------------------------------------------
item_pickup_device := class<final_super>(component):

    <#
        Calibrates the system to ensure good usage of the current limit
        for NPCs, making sure no persistent data is lost in the process.
    #>
    Calibrate():void =
        Logger.Print("Calibrate for {CurrentEnemies}")
```

## [Architecture] Architectural decisions

> Architecture is the set of decisions that are hardest to change later. Every abstraction must justify its complexity.

### Domain boundaries — keep modules cohesive

Group code by **domain**, not by technical layer. In a UEFN game:

```
Plugins/MyGame/Content/
├── Combat/        ← weapons, damage, hit detection
├── UI/            ← HUD, menus, widgets
├── Audio/         ← music, sfx
├── Persistence/   ← save data, leaderboards
├── Gameplay/      ← rounds, objectives, win conditions
└── Devices/       ← reusable device components
```

NOT:
```
Plugins/MyGame/Content/
├── Devices/       ← all devices, regardless of domain
├── Components/    ← all components
├── Managers/      ← all managers
```

Domain-grouped folders make ownership clear and dependencies natural.

### Dependency direction

Modules should depend **inward** on stable domains, not outward on volatile UI:

```
✅ Combat depends on Devices       (Devices is stable infrastructure)
✅ UI depends on Combat            (UI displays Combat state)
❌ Combat depends on UI            (Combat shouldn't know about HUD existence)
```

If you find yourself importing UI from Combat, that's a smell. Use events: Combat emits `PlayerDamagedEvent`, UI subscribes.

### Manager patterns — centralize, don't proliferate

```verse
# ❌ BAD: every entity its own loop, its own state
enemy_component := class(component):
    Update()<suspends>:void =
        loop:
            Sleep(0.1)
            UpdateEnemyLogic()

# ✅ GOOD: one centralized manager iterates over a list
enemy_manager_device := class(creative_device):
    ActiveEnemies : []enemy_component = array{}
    UpdateLoop()<suspends>:void =
        loop:
            Sleep(0.1)
            for (Enemy : ActiveEnemies):
                Enemy.UpdateLogic()
```

This pattern shows up in the [Concurrency] section too — it's both an architectural and a performance decision.

### Justify abstractions

Don't introduce a class, interface, or layer without a concrete reason. Pre-emptive abstractions are a common anti-pattern. Rule of thumb: **introduce abstraction at the second use case, not the first**. (DRY is a guideline, not a law — sometimes duplication is cheaper than the wrong abstraction.)

```verse
# ❌ BAD: speculative interface for "future flexibility"
weapon_interface := interface:
    Fire():void
    Reload():void
    Cooldown : float

# Only used by ONE class. Premature abstraction.

# ✅ GOOD: just the concrete class until you have 2+ implementations
rifle_device := class(creative_device):
    Fire():void = ...
    Reload():void = ...
    Cooldown : float = 0.5
```

### Reversibility over optimality

Prefer decisions that are easy to change. A "perfect" architecture you regret is worse than a "good enough" one you can refactor.

If a decision is hard to reverse (data schema, public API, persistent format), get it right the first time. If it's easy to change (internal naming, file organization), don't agonize.

## [Concurrency] Concurrency and temporal flow control

Verse's concurrency primitives are powerful but easy to misuse — performance issues, runaway tasks, memory leaks. Always design with **lifecycle, ownership, and execution frequency** in mind.

### Prefer events over polling

```verse
# ❌ BAD: permanent polling loop
UpdateLoop()<suspends>:void =
    loop:
        Sleep(0.0)
        CheckPlayers()
        CheckObjectives()

# ✅ GOOD: event-driven
OnPlayerSpawned(Player : player):void =
    InitializePlayer(Player)
```

### Every long-running task needs a clear owner and lifetime

```verse
# ❌ BAD: infinite loop without lifecycle validation
TrackPlayer(Player : player)<suspends>:void =
    loop:
        Sleep(1.0)
        UpdatePlayerUI(Player)

# ✅ GOOD: explicit lifetime check
TrackPlayer(Player : player)<suspends>:void =
    loop:
        Sleep(1.0)
        if (not Player.IsActive[]):
            break
        UpdatePlayerUI(Player)
```

### Use `spawn` carefully

`spawn` creates detached execution. Excessive or uncontrolled spawning creates race conditions and unpredictable order.

```verse
# ❌ BAD: many spawns on a frequent event
OnPlayerDamaged(Player : player):void =
    for (X : Numbers):
        spawn: PlayHitSound()
        spawn: PlayHitEffect()

# ✅ GOOD: spawn long-running independent systems
OnMatchStarted():void =
    spawn: MatchTimerLoop()
```

### Centralize update loops

See [Architecture] Manager patterns section above.

### Prefer larger sleep intervals over `Sleep(0.0)`

Ask: does this REALLY need to update every frame, or every second?

```verse
# ❌ BAD: busy waiting every frame
WaitUntilReady()<suspends>:void =
    loop:
        Sleep(0.0)
        if (IsReady?): break

# ✅ GOOD: 1-second polling
UpdateScoreboard()<suspends>:void =
    loop:
        Sleep(1.0)
        RefreshScoreboard()
```

### Loops must yield execution

A concurrent loop must always either wait for an event or use `Sleep`.

```verse
# ❌ BAD: aggressive loop
BadLoop()<suspends>:void =
    loop:
        UpdateEnemies()
        UpdateUI()

# ✅ GOOD: yield each iteration
GoodLoop()<suspends>:void =
    loop:
        Sleep(0.0)
        UpdateEnemies()
        UpdateUI()
```

### Use `race:` for timeouts

```verse
WaitForInteraction(Player : player)<suspends>:void =
    race:
        block:
            Player.InteractedWithButtonEvent.Await()
        block:
            Sleep(10.0)
            Print("Interaction timeout")
```

When `race:` cancels a branch, that branch's remaining code does NOT execute. Don't rely on cleanup logic in cancelled branches.

### Avoid concurrent mutation of shared state

```verse
# ❌ BAD: race condition
RaceCondition()<suspends>:void =
    race:
        block: set Score += 10
        block: set Score += 20

# ✅ GOOD: single owner, communicate intent
AddScore(Amount : int)<transacts>:void =
    set Score += Amount
```

### Validate references after suspension

```verse
# ❌ BAD: stale player reference
DelayedReward(Player : player)<suspends>:void =
    Sleep(10.0)
    GrantGold(Player)

# ✅ GOOD: revalidate
DelayedReward(Player : player)<suspends>:void =
    Sleep(10.0)
    if (Player.IsActive[]):
        GrantGold(Player)
```

## [Arrays] Handling large arrays

Verse passes arrays **by value**. Large arrays in hot paths cause memory pressure.

### Use `for` expressions for construction

```verse
# ❌ BAD: N allocations
BadArrayConstruction():void =
    var Result : []int = array{}
    for (I := 0..99):
        set Result = Result + array{I}

# ✅ GOOD: single allocation
GoodArrayConstruction():void =
    Result := for (Element := 0..99) do Element
```

### Cache `.Length` for repeated reads

```verse
# ✅ GOOD
GoodLengthCheck(Data : []int):void =
    Count := Data.Length
    var I : int = 0
    loop:
        if (I >= Count): break
        set I += 1
```

### Wrap large arrays in classes

```verse
# ✅ GOOD: wrapper class passed by reference
large_data_container := class<final>:
    Data : []int = array{}
    CachedCount : int = 0
```

### Combine arrays with `for`

```verse
# ✅ GOOD: single for-expression with multiple iterators
GetSpawnedAgents()<transacts>:[]agent =
    for (Spawner : Spawners, Agent : Spawner.GetAgents()):
        Agent
```

## [Expressions] Everything is a value

```verse
# ✅ if returns a value
GetLabel(Score : int) : string =
    if (Score > 100) then "High Score" else "Low Score"

# ✅ for returns an array
DoubleValues(Input : []int) : []int =
    for (X : Input) do X * 2

# ✅ for with inline filter
ResultAfter := for (X : Numbers, Mod[X, 2] = 0) do X * 2

# ✅ case returns a value
LabelAfter := case (CurrentDifficulty):
    game_difficulty.Easy   => "Beginner"
    game_difficulty.Normal => "Standard"
    game_difficulty.Hard   => "Expert"

# ✅ block returns its last value
CalculateAfter(Base : int) : int =
    Result := block:
        TempA := Base * 2
        TempB := TempA + 5
        TempB * 3
    Result
```

Prefer expression-oriented style over imperative `var` accumulation.

## [Effects] Effect specifiers

| Effect | When to use |
|---|---|
| `<computes>` | Pure computation. No reads, no mutations, no fail, no suspend. |
| `<transacts>` | Modifies mutable state with `set`. Required for rollback. |
| `<decides>` | Function can fail. Must be called with `[]`. |
| `<suspends>` | Suspends execution over time. |

Combinations: `<decides><transacts>` for state changes that may fail.

```verse
# Pure
GetDoubleValue(Value : int)<computes>:int = Value * 2

# Mutation
AddScore(Amount : int)<transacts>:void = set Score += Amount

# Mutating + failable
TrySpendScore(Cost : int)<decides><transacts>:void =
    Score >= Cost
    set Score -= Cost

# Temporal
StartCountdown()<suspends>:void =
    Sleep(1.0)
    Print("Countdown finished")
```

**Never add an effect just to silence the compiler.** If a function requires an effect, the function name and responsibility should make that effect obvious.

## [Failure] Failure-based control flow

```verse
# ❌ BAD: manual nested validation
GetPlayerWeapon(Player : player):?weapon =
    if (Player.IsActive[]):
        if (Weapon := WeaponPerPlayer[Player]):
            return option{Weapon}
    false

# ✅ GOOD: failure-based flow
GetPlayerWeapon(Player : player):?weapon =
    if:
        Player.IsActive[]
        Weapon := WeaponPerPlayer[Player]
    then:
        option{Weapon}

# ✅ GOOD: chained validation in <decides>
ActivateWeapon(Player : player)<decides><transacts>:void =
    Weapon := WeaponPerPlayer[Player]
    Weapon.IsEnabled[]
    Weapon.Activate[]

# ✅ GOOD: failable access over manual length check
GetFirstPlayer(Players : []player):?player =
    if (Player := Players[0]):
        option{Player}
```

## [Transactions] Automatic rollback

```verse
# ❌ BAD: manual revert
TryBuyItemBad(Player, Cost):logic =
    if (Stats := PlayerStatsMap[Player]):
        if (Stats.Coins >= Cost):
            set Stats.Coins -= Cost
            if (GrantItemToPlayer[]): return true
            set Stats.Coins += Cost  # manual rollback
    false

# ✅ GOOD: let the engine roll back
TryBuyItem(Player : player, Cost : int)<decides><transacts>:void =
    Stats := PlayerStatsMap[Player]
    Stats.Coins >= Cost
    set Stats.Coins -= Cost
    GrantItemToPlayer[]
```

**Side effects that DON'T roll back**: `Print()`, audio, VFX, async operations. Rollback only affects transactional mutable state.

## [Persistence] Correct use of persistent data

```verse
# ❌ BAD: stale player after delay
DelayedSave(Player : player)<suspends>:void =
    Sleep(5.0)
    SavePlayerStats(Player)

# ✅ GOOD: validate
DelayedSave(Player : player)<suspends>:void =
    Sleep(5.0)
    if (Player.IsActive[]):
        SavePlayerStats(Player)
```

### Prefer persistable classes over structs

Classes can be extended with new fields with default values — safer for future migrations.

```verse
player_stats := class<final><persistable>:
    Version : int = 1
    Coins : int = 0
    XP : int = 0
    Unlocks : []int = array{}
```

### Don't save on every change

Use a runtime cache + flush at controlled moments (autosave interval, player leaves, match ends, important purchases).

```verse
# ✅ GOOD: dirty flag + periodic save
AddCoins(Player : player, Amount : int)<transacts>:void =
    set Coins += Amount
    set StatsAreDirty = true

PeriodicSaveLoop()<suspends>:void =
    loop:
        Sleep(SaveInterval)
        for (Player -> Data : SessionCache):
            if (Player.IsActive[]):
                if (set PersistentStatsMap[Player] = Data):
                    Print("Saved.")
```

## [Security] Trust boundaries and input validation

> The client is the enemy. Anything that comes from a player's input — a button press, a trigger, a stat increase — should be validated by the device, not trusted.

### Validate input from players

```verse
# ❌ BAD: trust the input
OnDamageButtonPressed(Player : player, Damage : int):void =
    set Player.Health -= Damage  # what if Damage is INT_MAX?

# ✅ GOOD: validate range
OnDamageButtonPressed(Player : player, Damage : int):void =
    ClampedDamage := if (Damage < 0) then 0 else if (Damage > MaxAllowedDamage) then MaxAllowedDamage else Damage
    set Player.Health -= ClampedDamage
```

### Don't expose unbounded operations to player triggers

```verse
# ❌ BAD: trigger spawns unbounded loop
OnPlayerInteract(Player : player):void =
    spawn: SpawnLoop()  # if this loop never ends, leaks

# ✅ GOOD: bounded, with cancellation
OnPlayerInteract(Player : player):void =
    spawn:
        race:
            block: SpawnLoop()
            block:
                Sleep(MaxSpawnDuration)
                Print("Spawn timed out")
```

### Verify ownership before mutating player data

```verse
# ❌ BAD: any player can modify any other player's stats
GivePoints(Sender : player, Recipient : player, Amount : int)<transacts>:void =
    if (Stats := PlayerStatsMap[Recipient]):
        set Stats.Points += Amount

# ✅ GOOD: validate sender has authority
GivePoints(Sender : player, Recipient : player, Amount : int)<decides><transacts>:void =
    Sender = Recipient or IsAdmin[Sender]
    Amount > 0
    Amount <= MaxTransfer
    Stats := PlayerStatsMap[Recipient]
    set Stats.Points += Amount
```

### Persistent data is forever

Persistent saves carry across sessions and published versions. **A bug that corrupts persistent data is much more expensive than a bug in transient state** — it can't be reverted by restarting. Validate before writing.

## [Correctness] Avoiding common bug classes

### Off-by-one prevention

```verse
# ❌ BAD: < length but indexes 0..length, off by one
for (I := 0..Players.Length):
    if (P := Players[I]):  # fails on last iteration
        Print("{P}")

# ✅ GOOD: range stops one before length
for (I := 0..Players.Length - 1):
    if (P := Players[I]):
        Print("{P}")

# ✅ BETTER: iterate the array directly
for (P : Players):
    Print("{P}")
```

### Always handle the failure case in `<decides>`

```verse
# ❌ BAD: ignoring failure silently
DoSomething(Player : player):void =
    if (Stats := PlayerStatsMap[Player]):
        set Stats.Score += 1
    # what if PlayerStatsMap doesn't have Player? silently nothing.

# ✅ GOOD: explicit handling or propagation
DoSomething(Player : player)<decides>:void =
    Stats := PlayerStatsMap[Player]  # propagate failure to caller
    set Stats.Score += 1
```

### State invariants

If your class has rules that always hold (e.g. "Players list is never empty after OnBegin"), document them at the top of the class. The AI / human reviewer can verify changes don't break the invariants.

```verse
#-----------------------------------------------------------------------------
#   round_manager
#
#   Invariants:
#   - ActivePlayers.Length >= 1 once OnBegin completes
#   - CurrentRound is in [1, MaxRounds]
#   - WinCondition is non-empty
#-----------------------------------------------------------------------------
round_manager := class(creative_device):
    ...
```

## [Maintainability] Code that survives 6 months

> Will the next person who reads this code understand it without asking the author? If not, fix it now.

### Function size targets

| Lines | Verdict |
|---|---|
| 1–20 | Ideal. One responsibility, easy to test mentally. |
| 21–50 | OK if the function is genuinely doing one thing. |
| 51–100 | Smell. Probably has multiple responsibilities. Consider splitting. |
| 100+ | Almost always wrong. Extract sub-functions. |

### Cyclomatic complexity awareness

If your function has more than 3 levels of nested `if`/`for`, extract some of it. Deep nesting is a sign that the function is doing too much.

```verse
# ❌ BAD: 5 levels deep
ProcessThing(Thing):void =
    if (Thing.IsActive?):
        if (Thing.HasOwner?):
            for (Item : Thing.Items):
                if (Item.IsValid?):
                    for (Sub : Item.Children):
                        DoSomething(Sub)

# ✅ GOOD: extract the inner work
ProcessThing(Thing):void =
    if:
        Thing.IsActive?
        Thing.HasOwner?
    then:
        for (Item : Thing.Items, Item.IsValid?):
            ProcessItem(Item)

ProcessItem(Item):void =
    for (Sub : Item.Children):
        DoSomething(Sub)
```

### Self-documenting code over comments

```verse
# ❌ BAD: needs a comment because the logic is opaque
# checks if player can transition to next round
if (P.S = 3 and P.R < M and not P.E):
    Advance()

# ✅ GOOD: extracted intent is self-evident
if (CanAdvanceToNextRound[P]):
    Advance()

CanAdvanceToNextRound(P : player)<decides>:void =
    P.State = player_state.Ready
    P.CurrentRound < MaxRounds
    not P.IsEliminated
```

## [Testing] What and how to verify

UEFN doesn't have unit-test infrastructure like other languages, so testing is mostly manual. The discipline still matters.

### Required testing per change type

| Change type | Minimum test |
|---|---|
| Bug fix | Reproduce the bug, confirm fix, then test surrounding cases |
| New feature | Happy path + at least 2 edge cases (empty, max, invalid input) |
| Refactor | Old test cases still pass; performance comparable |
| Hotfix | Smoke test the affected code path under realistic load |

### Edge cases to consider for any player-touching code

- Player joins mid-action
- Player leaves mid-action
- Two players triggering simultaneously
- Network latency (test in actual UEFN session, not just editor)
- Empty/full collections
- Min/max values for inputs

### "Tested in" field discipline

The Slack review template has a "Tested in" field. **Don't lie.** "UEFN Session" means you actually launched a session and tested. "UEFN Editor" means you only tested in the editor (less reliable). "Manual" means you reasoned about it without testing — only acceptable for trivial changes.

## [Enums] Enum size

Restrict enums to **≤ 255 elements**. We've observed sessions failing to launch with larger enums. Split into multiple enums if you need more.

---

## [WeakMaps] Proper use of weak maps

Use weak maps only for:
- **Persistent data**: `var PersistentStatsMap : weak_map(player, player_stats) = map{}`
- **Session-keyed singletons** (manager pattern): `var ServiceMap : weak_map(session, my_manager) = map{}`

**DON'T** use weak maps with `player` as key for non-persistent data — they don't reliably free entries when players leave. Use a regular `map` with manual cleanup on `OnPlayerRemoved`.

## [Events] Events and event handlers

```verse
# ✅ GOOD
PlayerConnectedEvent : event(player) = event(player){}    # event itself
OnPlayerConnected(Player : player):void = ...             # handler

PlayerConnectedEvent.Subscribe(OnPlayerConnected)
```

Event names: NO `On` prefix. Handler names: `On` prefix. This eliminates collisions in the same class.

## Sections quick reference

When the AI cites a violation, it uses these tags with severity emojis:

- `[Language]` — English-only rule
- `[Naming]` — types, members, booleans, events
- `[Formatting]` — indentation, operator spacing
- `[Separators]` — visual separators between methods
- `[Comments]` — header, descriptive, multi-line styles
- `[Architecture]` — domain boundaries, dependency direction, abstraction justification
- `[Concurrency]` — events vs polling, spawn, race, lifecycle, suspension
- `[Arrays]` — for-expressions, length caching, wrapper classes
- `[Expressions]` — value-oriented patterns
- `[Effects]` — `<computes>`, `<transacts>`, `<decides>`, `<suspends>`
- `[Failure]` — failure-based flow, sequential validation
- `[Transactions]` — automatic rollback, atomicity
- `[Persistence]` — player validation, classes over structs, save discipline
- `[Security]` — input validation, ownership checks, persistent data integrity
- `[Correctness]` — off-by-one, failure handling, state invariants
- `[Maintainability]` — function size, complexity, self-documentation
- `[Testing]` — what to test per change type
- `[Enums]` — size limit
- `[WeakMaps]` — when allowed, when forbidden
- `[Events]` — event vs handler naming
